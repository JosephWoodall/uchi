# Architecture

> **Python users:** you do not interact with the architecture directly. Use
> `from uchi import Uchi` and call `learn()` / `ask()`. See [Python API →](python-api.md).

---

Uchi is built on one principle: **a fallible proposer plus a reality-anchored
verifier.** It generates candidate answers (and reasoning steps) freely, then
*verifies them against what the brain actually knows* before speaking — and
abstains when it can't. No LLM, no knowledge in neural weights.

## The three lanes behind `ask()`

Every natural-language message is classified by `intent_router` and routed:

```
                 ┌─────────── skill ───────────►  SkillRegistry  (/classify, /forecast, code)
ask(str) ─► router ── social ──────────────────►  ConversationEngine  (free chit-chat, no oracle)
                 └─────────── factual ──────────►  Generate-and-Ground  (grounded, or abstain)
                                                        │
                                       multi-step? ─────┴──► ReasoningEngine (verified steps)
```

Only the **factual** lane is verified. Social replies assert no facts, so they are
free-generated by a small dialogue model — that is how Uchi has a personality
without weakening its honesty on facts.

## The factual lane — Generate-and-Ground

| Stage | Component | Role |
|-------|-----------|------|
| retrieve | `retrieval.SemanticIndex` | skip-gram passage index over the brain; hybrid lexical+semantic ranking |
| generate | `decoder.NeuralDecoder` | small from-scratch retrieval-conditioned seq2seq (extractive fallback) |
| verify | `oracle.FactCheckOracle` + `answerability.AnswerabilityChecker` | is the answer supported, and does the evidence actually answer the question? |
| emit / abstain | `generate_and_ground.GenerateAndGround` | grounded answer, else "I don't have grounded knowledge to answer that." |

Generalisation comes from generating over retrieved knowledge; honesty comes from
verifying before speaking. See [Generate-and-Ground →](generate-and-ground.md).

## Reasoning — verified steps

Multi-step questions go through `reasoning.ReasoningEngine`: decompose the goal,
execute each step with an operator that can *verify its own output* (a symbolic
math oracle, a REPL-checked code operator, or grounded factual lookup), compose the
verified results, and **abstain naming the step it couldn't establish**. Every
emitted conclusion is a chain of verified steps. See [Reasoning →](reasoning.md).

## What the trie is (and isn't)

The credibility-weighted prefix trie (`predictor` / `SequenceGenerator`) is Uchi's
**recall and grounding** layer — append-only, so it never forgets. It is *not* the
answer generator (its raw generation is incoherent; that is why the neural decoder
exists). It stores knowledge; Generate-and-Ground retrieves, generates, and verifies
over it.

## Skills

`SkillRegistry` exposes analytical capabilities (`tabular`, `timeseries`, forecasting,
anomaly detection, code) as callable skills, and is the socket for future
verified-reasoning operators.

## Persistence & compounding

`ask()` returns a string; `learn()` accepts one. `learn()` feeds both the trie and
the retrieval index live, so knowledge added to one instance grounds another. The
brain (`brain.uchi`) is a gzip-pickled router carrying the trie + semantic index.

## Retired

The former SSM value-head QA path, GRPO training, the convergent MCTS engine, and
trie-based text generation were removed. The trie is kept for recall/grounding; all
generation is now the neural decoder gated by the oracle.
